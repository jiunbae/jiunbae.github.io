# 브랜딩이 싫어서 브랜딩 도구를 만들었습니다: Kurim 이야기

저는 사이드 프로젝트를 꽤 많이 만듭니다. 문제는 코드가 아니라 그다음이에요. 기능을 다 만들어놓고 나면 항상 같은 벽에 부딪힙니다.

> "이거 이름 뭐라고 하지? 로고는? 랜딩 페이지 문구는?"

솔직히 이 단계가 제일 싫었습니다. 개발은 반나절이면 되는데 브랜딩에서 며칠이 날아갑니다. Figma를 열었다가 로고 하나 못 만들고 닫은 게 몇 번인지 모르겠어요. 그래서 대부분의 프로젝트는 임시 이름에 시스템 폰트 그대로, 로고는 이모지 하나로 때우고 배포됐습니다.

그러다 문득 이런 생각이 들었어요. "브랜딩이 그렇게 싫으면, 브랜딩을 대신 해주는 걸 만들면 되잖아?" 그렇게 시작한 게 [Kurim](https://kurim.jiun.dev)입니다. 랜딩 페이지 첫 문장에 그대로 써놨어요. **"브랜딩을 싫어하는 개발자를 위해 만들었습니다."**

## 뭘 하는 도구인가

한 줄로 말하면, 대충 던진 아이디어를 완성된 브랜드 한 세트로 바꿔주는 도구입니다. 로고, 브랜드 아이덴티티 문서, 랜딩 페이지, 컬러 팔레트까지 한 번에 나옵니다.

![Kurim 랜딩 페이지 상단. '브랜딩을 싫어하는 개발자를 위해 만들었습니다'라는 문구 아래 '10분 만에 완성하는 브랜드'라는 큰 제목과 'AI가 로고, 브랜드 아이덴티티, 랜딩 페이지를 한 번에 생성합니다'라는 설명, 무료로 시작하기·데모 보기 버튼이 보인다.](/images/posts/kurim-ai-branding/01-hero.png)

"10분 만에 완성하는 브랜드"라고 써놨는데, 처음엔 좀 과장인가 싶었지만 실제로 프로젝트 설명 입력하고 커피 한 잔 내려오는 동안 결과물이 나오긴 합니다. 물론 그 결과물을 그대로 쓸지는 또 다른 문제고요.

흐름은 딱 세 단계입니다. 프로젝트를 설명하고, AI가 전부 생성하고, 원클릭으로 배포하는 것.

![Kurim 사용 방법과 기능 소개. 위쪽에는 프로젝트 설명, AI가 모든 것 생성, 원클릭 배포의 3단계가 번호와 함께 있고, 아래쪽에는 AI 로고 생성, 브랜드 아이덴티티 문서, 원클릭 배포, 컬러 & 타이포그래피, 브랜드 킷 내보내기, 개발자 API의 6개 기능 카드가 격자로 배열되어 있다.](/images/posts/kurim-ai-branding/02-workflow-features.png)

개발자를 대상으로 만든 티가 나는 부분이 몇 개 있는데, 저는 이런 걸 넣을 때가 제일 재밌었어요.

하나는 브랜드 문서를 `BRAND.md`, `README.md` 같은 마크다운으로 뽑아준다는 점입니다. 브랜드 가이드라인을 PDF로 받아봤자 저는 안 읽거든요. 그런데 레포에 `BRAND.md`로 들어와 있으면 얘기가 다르죠.

또 하나는 GitHub README 자동 입력입니다. 프로젝트 설명을 처음부터 타이핑하기 귀찮으니까, 이미 있는 GitHub 레포 주소를 넣으면 README를 읽어서 프로젝트 정보를 알아서 채워줍니다. 이건 만들면서 "아 이거 나부터 쓰겠다" 싶었던 기능이에요.

그리고 결과물을 GitHub Pages나 Vercel로 바로 배포하거나, 로고와 가이드라인을 SVG/PNG zip으로 내보내고, REST API랑 웹훅으로 생성 자체를 워크플로우에 끼워 넣을 수도 있습니다.

## 이미 이런 거 있지 않나?

만들기 전에 저도 이 생각을 제일 먼저 했습니다. Looka나 Brandmark 같은 AI 로고/브랜딩 서비스는 이미 유명하고, 결과물도 좋아요. 실제로 만들기 전에 경쟁 서비스를 한참 뜯어봤습니다.

그런데 그 도구들은 대체로 "사업하는 사람"을 위한 물건이더라고요. 회사명 넣고, 스타일 고르고, 예쁜 로고랑 명함 시안 받고, 브랜드 키트를 구독하는 흐름입니다. 좋습니다. 근데 저 같은 개발자한테는 미묘하게 안 맞았어요. 결과물이 마크다운으로 안 나오고, API가 없고, 만든 걸 git에 커밋해서 배포하는 흐름이랑 따로 놀았습니다.

그래서 Kurim은 처음부터 개발자 워크플로우에 붙는 걸 목표로 잡았습니다. 브랜드 문서는 `.md`로, 자산은 zip으로, 생성은 API로. 로고 자체의 품질로 경쟁하겠다는 게 아니라, "개발자가 이미 쓰는 도구 사이에 자연스럽게 끼는 브랜딩"을 노린 거죠.

## 로고 생성의 현실

로고는 DALL·E 3로 생성합니다. 그런데 이미지 생성 모델로 로고를 만들어본 분들은 이미 웃고 계실 것 같아요. 결과물 자체는 꽤 그럴듯하게 나옵니다.

![Kurim이 생성한 로고 시안 세 개. 왼쪽은 미니멀한 워드마크 스타일, 가운데는 보라색 계열의 기하학적 아이콘, 오른쪽은 역동적인 레터폼 형태의 로고로, 흰 배경 위에 나란히 배치되어 있다.](/images/posts/kurim-ai-branding/03-logo-samples.png)

가운데 기하학적 아이콘이나 오른쪽 레터폼은 그대로 써도 될 만큼 괜찮게 나왔어요. 그런데 왼쪽 워드마크를 자세히 보면 문제가 보입니다. "kurim"이라고 써야 하는데 "KURIN KIURIM" 비슷한 정체불명의 글자가 박혀 있어요.

이게 이미지 생성 모델의 고질병입니다. 글자를 그림처럼 그리기 때문에 철자를 못 맞춰요. 그래서 텍스트가 들어가는 워드마크 계열은 아직 사람 손이 필요하고, Kurim도 이 부분은 솔직히 완전히 해결하지 못했습니다. 심볼이나 추상적인 아이콘은 잘 나오는데, 브랜드명을 정확히 렌더링하는 건 여전히 복불복이에요. 컨셉을 빠르게 여러 개 뽑아보는 용도로는 충분하지만, "최종 로고 확정"까지 자동으로 가기엔 아직 멀었다는 게 솔직한 평가입니다.

## 기술 구성

프런트엔드는 Next.js 14(App Router) + TypeScript + Tailwind + shadcn/ui로 짰습니다. 다국어는 next-intl로 한국어/영어를 지원해요. 백엔드는 FastAPI + SQLAlchemy 2.0 + Pydantic v2고, DB는 PostgreSQL 16, 캐시랑 레이트 리밋은 Redis 7을 씁니다. AI는 OpenAI(GPT-4로 카피/브랜드 문서, DALL·E 3로 로고)에 Replicate를 섞었고요.

프런트/백엔드를 나눈 이유는 단순합니다. 로고 생성이 오래 걸리는 비동기 작업이라, 요청을 던져놓고 WebSocket으로 진행 상황을 실시간으로 받아야 했거든요. Next.js 안에서 다 처리할 수도 있었겠지만, 무거운 생성 작업은 FastAPI 쪽 백그라운드로 빼는 게 마음이 편했습니다.

배포는 Kubernetes에 올려서 Gitea Actions로 CI/CD를 돌리고, HPA로 오토스케일링까지 붙였습니다. 사이드 프로젝트치고 과한 거 아니냐고 하면... 맞아요. 근데 어차피 클러스터는 이미 돌아가고 있었고, 새 서비스 하나 얹는 김에 제대로 해두자는 마음이었습니다.

참고로 이 프로젝트는 하루 이틀 만에 뚝딱 나온 게 아닙니다. 첫 커밋이 2025년 11월이었고, 처음엔 그냥 로고 생성기 하나였어요. 거기에 브랜드 문서, 랜딩 페이지, 배포, 결제, 팀 기능이 몇 달에 걸쳐 하나씩 붙었습니다. 인증도 이메일 인증에 Google·GitHub OAuth까지 들어갔고요. "로고 하나 만들어주는 거 아니었어?" 싶겠지만, 라이브 서비스처럼 굴리려다 보니 어느새 이만큼 커져 있었습니다.

## 사이드 프로젝트인데 보안 감사를 배웠다

이 프로젝트에서 가장 예상 밖이었던 건 시간을 어디에 제일 많이 썼느냐는 거예요. 로고 품질도 아니고, UX도 아니고, **보안**이었습니다.

커밋 로그를 보면 `R01`부터 `R21`까지, 스무 번이 넘는 리뷰 라운드의 흔적이 남아 있어요. OAuth CSRF, SSRF 하드닝, JWT 해싱, Redis 커넥션 정리, XSS, 랜딩 페이지 iframe 샌드박싱, 웹훅 SSRF... 브랜딩 도구를 만들려고 시작했는데 정작 배운 건 프로덕션 보안이었습니다.

특히 SSRF가 골치였어요. GitHub README를 읽어오거나 사용자가 준 URL을 처리하는 기능이 여기저기 있다 보니, 서버가 내부 네트워크를 긁어오지 않게 막는 데 신경을 많이 썼습니다. DNS 리바인딩까지 고려해서 막다 보면 "내가 지금 로고 만드는 앱을 짜는 게 맞나" 싶은 순간이 옵니다.

여기에 의존성 CVE도 끊임없이 터졌습니다. Pillow, cryptography, starlette, Next.js... 커밋 로그 절반이 `fix(deps)`예요. 라이브 서비스를 유지한다는 게 기능을 만드는 것보다 이런 뒤치다꺼리에 가깝다는 걸 새삼 느꼈습니다.

## 아직 부족한 것

솔직하게 적자면, Kurim은 아직 "완성된 제품"이라기보단 "잘 돌아가는 실험"에 가깝습니다.

로고 텍스트 문제는 앞서 말한 대로 남아 있고, 브랜드 탐색(Explore) 페이지는 아직 공개된 프로젝트가 없어서 텅 비어 있습니다. 결제(Stripe)나 팀 협업, 요금제 같은 기능은 다 붙여놨지만 실사용자가 몰려서 검증된 상태는 아니에요. 기능의 완성도와 실사용의 검증은 다른 문제니까요.

그럼에도 만들길 잘했다고 생각하는 이유는, 적어도 저부터 이제 새 프로젝트를 시작할 때 이름과 로고 앞에서 며칠씩 멈추지 않게 됐다는 거예요. 완벽한 로고는 아니어도, "일단 이걸로 시작하고 나중에 다듬자"라고 넘어갈 수 있는 무언가가 생겼습니다. 사이드 프로젝트에서 브랜딩이 병목이었던 사람에게는 그것만으로도 충분한 것 같아요.

궁금하신 분은 [kurim.jiun.dev](https://kurim.jiun.dev)에서 직접 던져보셔도 됩니다. 여러분의 프로젝트 이름은 제 것보단 제대로 렌더링되길 바랍니다.
