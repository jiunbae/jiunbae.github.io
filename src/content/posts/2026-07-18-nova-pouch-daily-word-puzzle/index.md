김초엽의 《양면의 조개껍데기》에 〈비구름을 따라서〉라는 단편이 있다. 거기 이런 문장이 나온다.

> 막을 건너온 사소한 물건들, 그 너머에 존재하는 것은 거대한 세계와 사람들.
> 할 수 있는 것은 오직 상상하는 일뿐.

이 문장을 읽고 한참 멍때렸다. 어딘가에서 넘어온 사소한 물건 하나로 그 너머의 세계를 통째로 상상한다는 발상이 좋았다. 그래서 그걸 매일 하는 작은 웹 게임으로 만들었다. 이름은 [Nova Pouch](https://nova-pouch.jiun.dev), 우리말로는 차원의 주머니다.

![Nova Pouch 첫 화면. 차원의 주머니 일러스트와 김초엽의 인용 문장, 시작하기 버튼이 보인다.](/images/posts/nova-pouch-daily-word-puzzle/home.png)

## 파편 세 개, 세계 하나

규칙은 단순하다. 주머니가 세 개 있다.

```
🔴 물건(물질계)   우산, 신발, 시계, 열쇠, 베개…
🔵 속성(성질계)   투명한, 날아다니는, 노래하는, 감정을 읽는…
🟢 제약(규율계)   물에 녹는, 밤에만 작동하는, 한 번만 쓸 수 있는…
```

각 주머니에서 파편을 하나씩 뽑으면 "밤에만 작동하는 / 노래하는 / 시계" 같은 조합이 나온다. 그리고 미션은 이 이상한 조합이 그냥 당연한 일상인 세계를 상상해서 짧게 글로 쓰는 것이다. 시계가 노래로 시간을 알려주고 밤에만 움직이는 도시는 어떤 곳일까. 그 세계에 사는 사람들은 아침을 어떻게 맞을까. 정답은 없고, 남이 쓴 걸 보면 같은 조합인데도 전혀 다른 세계가 나온다.

![주머니를 여는 화면. 빨간 주머니를 탭하면 파편이 나타난다는 안내와 주머니 버튼이 있다.](/images/posts/nova-pouch-daily-word-puzzle/draw.png)

![세 파편이 합쳐진 화면. 젓가락·진동하는·한 손으로만 쓸 수 있는이 쌓여 "한 손으로만 쓸 수 있는 진동하는 젓가락" 한 문장이 된다.](/images/posts/nova-pouch-daily-word-puzzle/review.png)

파편은 지금 세 주머니 합쳐서 158개다(물건 53, 속성 54, 제약 51). 처음엔 훨씬 적었는데 쓰다 보니 계속 늘었다. 코드 주석에는 아직도 "50-token registry"라고 적혀 있는데, 그건 옛날에 붙여둔 이름이고 실제로는 세 배쯤 불어난 셈이다. 이런 게 개인 프로젝트의 정직한 흔적이라고 생각한다.

## 오늘의 파편은 모두에게 같아야 한다

데일리 퍼즐이니까 핵심은 "오늘 뽑히는 세 파편이 전 세계 모든 접속자에게 동일해야 한다"는 것이다. 서버가 매일 랜덤을 굴려서 내려주는 방법도 있지만, 그러면 서버가 죽으면 오늘의 퍼즐도 같이 죽는다. 그래서 날짜 문자열만 있으면 누구나 같은 결과를 계산해내는 결정론적 해시를 썼다.

```typescript
function hashWithPrime(input: string, prime: number): number {
  let hash = 0;
  for (const char of input) {
    hash = (hash + char.charCodeAt(0) * prime) % 2_147_483_647;
  }
  return hash;
}

// 날짜 + 주머니마다 다른 소수를 곱해서 서로 다른 인덱스를 뽑는다
red   = redPool[hashWithPrime(`${d}:red`, 31)     % redPool.length];
blue  = bluePool[hashWithPrime(`${d}:blue`, 37)   % bluePool.length];
green = greenPool[hashWithPrime(`${d}:green`, 41) % greenPool.length];
```

주머니마다 소수를 31, 37, 41로 다르게 준 건, 같은 날짜에 세 주머니가 우연히 같은 패턴으로 움직이지 않게 하려는 것이다. 날짜는 KST(UTC+9) 자정 기준으로 넘어간다. 한국 사람이 쓰는 서비스니까 자정이 넘어가는 감각도 한국 시간이어야 한다고 생각했다.

Day #1은 2025년 8월 27일이다. 아무 날이나 잡은 게 아니라 《양면의 조개껍데기》 초판 1쇄 발매일로 맞췄다. 게임을 켜면 "Day #NNN"이 뜨는데, 그 숫자의 0일이 그 책이라는 게 나만 아는 작은 이스터에그다.

데일리는 하루 한 번이다. 오늘 세계를 완성하면 `nova-pouch-daily-completion` 키에 완료 기록이 남고, 화면은 다음 퍼즐까지 남은 시간을 세는 카운트다운으로 바뀐다. 자정이 지나면 새 파편이 뜬다. "하루에 한 번, 짧게"라는 리듬이 이 게임의 전부라, 오늘 걸 끝냈으면 내일까지는 기다리게 만드는 편이 맞다고 봤다. 대신 더 놀고 싶은 사람을 위해 프리 플레이는 따로 무제한으로 열어뒀다.

### 서버가 나보다 뒤처졌던 이야기

원래는 데일리 파편을 서버에서 받아왔다. 서버도 똑같은 `hashWithPrime`을 돌리니까 결과가 같을 거라고 믿었는데, 어느 날 클라이언트에서 뽑히는 파편이랑 서버에서 내려주는 게 달라졌다. 파편 목록을 클라이언트 쪽에서 계속 늘리는 동안, 서버는 옛날 15개짜리 풀에 멈춰 있었던 것이다. 같은 해시 함수라도 풀 크기가 다르면 `% length` 결과가 어긋난다.

고민하다가 그냥 네트워크 경로를 껐다. 지금 daily 계산은 무조건 로컬 레지스트리에서 한다. 코드에도 "서버 풀이 최신으로 맞춰지면 다시 켜라"는 주석을 남겨뒀는데, 아직 안 켰다. 단일 소스가 하나뿐이면 어긋날 데가 없다는 게, 돌아보니 데일리 퍼즐 같은 것에는 오히려 안전한 선택이었다.

## 난이도는 별로 표시된다

파편마다 난이도가 1~3으로 붙어 있다. "우산"은 1이고 "시간을 되돌리는"은 3이다. 세 파편의 난이도 합을 별점으로 환산해서 Very Easy부터 Very Hard까지 다섯 단계로 보여준다. 합이 3 이하면 별 하나, 7 이상이면 별 다섯 개다.

프리 플레이(무제한 랜덤 뽑기)에서는 추첨 자체도 난이도를 반영한다. 가중치를 `4 - difficulty`로 줘서, 어려운 파편일수록 덜 나오게 했다. 매번 초고난도 조합만 나오면 글쓰기가 숙제가 되니까, 대체로는 만만한 걸 주고 가끔 어려운 걸 섞는 쪽이 놀기 좋았다.

## 프레임워크 없이 만들었다

이번엔 의도적으로 프레임워크를 안 썼다. React도, 상태관리 라이브러리도 없다. 런타임 의존성 제로, 순수 TypeScript + Vite다. DOM API만으로 렌더링하고, `innerHTML`은 아예 안 쓴다. 문자열로 HTML을 조립하지 않으니 사용자가 쓴 세계관 텍스트를 그대로 화면에 꽂아도 XSS가 날 구석이 없다. 커뮤니티 피드처럼 남이 쓴 글을 보여주는 화면이 있는 이상, 이건 편의보다 안전 문제였다.

상태는 직접 만든 pub/sub 상태머신으로 굴린다. `IDLE → DRAWING → REVIEW → WRITING → COMPLETE`를 오가고, 렌더러는 현재 상태를 읽어서 idempotent하게 다시 그린다. 주머니를 뽑을 때 흔들리고(shake) → 카드가 떠오르고(rise) → 뒤집히고(flip) → 텍스트가 뜨는(reveal) 애니메이션도 CSS transform과 Web Animations API로만 만들었다. 전부 합쳐 `src` 기준 5,600줄 정도인데, 프레임워크가 없으니 매 줄이 다 내가 쓴 줄이라 어디서 뭐가 도는지는 확실히 안다.

작은 인터랙션에도 신경을 좀 썼다. 뽑기는 모바일 오터치를 막으려고 `pointerup` 기준으로 확정하고, 애니메이션이 도는 동안은 버튼을 잠가서 연타로 상태가 꼬이는 걸 막는다. 마우스/터치/키보드(Space, Enter)를 Pointer Events로 한 갈래에 모아뒀고, 공개된 파편 텍스트는 `aria-live`로 스크린리더가 읽어준다. `prefers-reduced-motion`을 켠 사람에겐 애니메이션을 확 줄인다. 화려한 기능은 아니지만, 하루 한 번 잠깐 들어와서 파편을 뽑는 게임이라 그 잠깐이 매끄러운 게 제일 중요했다.

## 아키텍처를 두 번 갈아엎었다

솔직히 여기서 좀 헤맸다. 처음엔 GitHub Pages에 올라가는 정적 SPA였다. 그런데 "남이 쓴 세계를 읽는 커뮤니티 피드"랑 로그인이 붙으니 백엔드가 필요해졌고, Bun + Hono로 풀스택 앱을 만들어서 Kubernetes에 올렸다. OAuth를 다섯 개(구글, 깃허브, 카카오, 네이버, X)까지 붙였다가, 관리가 감당이 안 돼서 뺐다 넣었다를 반복했다. CI도 GitHub Actions에서 Gitea Actions로 옮기고, arm64/amd64 멀티아치 빌드까지 얹었다.

그러다 결국 마지막엔 다시 정적 SPA로 돌아왔다. Nova Pouch만을 위한 백엔드를 따로 운영하는 대신, API 호출은 내가 공용으로 쓰는 `jiun-api`로 넘겼다. 게임 본체는 어차피 로컬에서 다 도니까, 서버는 피드/기록 공유 같은 "있으면 좋은" 기능만 담당하면 됐다. 오프라인 우선(offline-first) 설계라 API가 죽어도 파편 뽑고 글 쓰고 저장하는 핵심 루프는 localStorage만으로 멀쩡히 돈다.

이 왕복이 뻘짓이었냐면, 반쯤은 맞다. 처음부터 "게임은 로컬, 서버는 옵션"으로 선을 그었으면 K8s까지 갔다 올 일은 없었다. 다만 한 바퀴 돌고 나서야 어디까지가 진짜 서버가 필요한 부분인지 몸으로 알게 된 것도 사실이다.

## 곁다리들

- **공유 카드**: 내가 쓴 세계를 Canvas 2D로 이미지 카드로 그려서 트위터나 기본 공유로 내보낸다. 텍스트만 링크로 보내는 것보다 파편 세 개가 박힌 카드가 훨씬 그럴듯해 보인다.
- **이중 언어**: `data-i18n` 속성 기반으로 한국어/영어를 전부 지원한다. 줄바꿈도 `\n`을 문자열 치환이 아니라 DOM 노드로 처리해서, 여기서도 innerHTML을 안 쓴다.
- **프리셋 세계**: 책 속 설정을 참조한 "녹색의 세계" 프리셋이 있는데, 존재하지 않는 날짜 4월 31일을 일부러 살려뒀다. 실재하지 않는 세계니까 달력에 없는 날이 어울린다.
- **글쓰기 진척 표시**: 파편을 하나씩 뽑을 때마다 유리구슬 모양 점이 채워지고, 뽑은 파편들이 스택으로 쌓이는 걸 보여준다. 별건 아닌데, 지금 몇 개 뽑았고 뭘 뽑았는지 한눈에 보이는 게 글 쓰기 전 몰입에 은근히 중요했다.
- **오프라인 복원**: 기록은 버전 필드를 달아서 저장한다. 스키마가 바뀌어서 복원에 실패하면 설정만 살리고 기록은 백업 후 초기화한다. 새로고침 한 번에 남이 쓴 글까지 날려먹지 않으려는 안전장치다.

## 마무리

기술적으로 대단한 프로젝트는 아니다. 해시 하나, 상태머신 하나, DOM 렌더링. 재미있었던 건 오히려 파편 목록을 채우는 일이었다. "감정을 읽는", "사용할 때마다 작아지는" 같은 조각을 하나씩 적으면서, 이게 다른 조각이랑 만나면 어떤 세계가 될까 혼자 상상하는 시간이 좋았다.

만들고 나서 며칠은 나도 그냥 유저로 접속해서 오늘의 파편을 뽑고 세 줄쯤 끄적인다. 정답이 없는 게 이 게임의 유일한 규칙이라, 매일 뽑히는 파편이 나한테도 매일 다른 낙서를 시킨다. 사소한 물건 하나로 그 너머의 세계를 상상하는 일 — 결국 원래 그 문장이 하려던 것도 이거였던 것 같다.

궁금하면 [nova-pouch.jiun.dev](https://nova-pouch.jiun.dev)에서 오늘의 파편을 한번 뽑아보시길.
