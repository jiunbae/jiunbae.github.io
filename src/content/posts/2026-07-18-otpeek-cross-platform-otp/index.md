# 2FA 코드 하나 보려고 폰을 꺼내는 게 싫어서 만든 OTPeek

## 들어가며

데스크톱 앞에 앉아서 로그인을 하는데 2FA 코드를 넣으라고 한다. 그럼 나는:

1. 주머니에서 폰을 꺼내고
2. 폰을 잠금 해제하고
3. 인증기 앱을 열고
4. 수십 개 계정 중에 해당 계정을 찾고
5. 6자리를 눈으로 읽고
6. 코드가 만료되기 전에 키보드로 옮겨 친다

이걸 하루에 몇 번씩 한다. 손 뻗으면 닿는 곳에 컴퓨터가 있는데, 굳이 폰을 꺼내서 눈으로 숫자를 읽고 다시 타이핑하는 이 왕복이 매번 거슬렸다. 코드는 이미 화면 어딘가에 항상 떠 있으면 되는 거 아닌가? 위젯 하나에 코드가 보이고, 탭 한 번에 복사되면 끝날 일이었다.

그래서 [OTPeek](https://jiun.dev/OTPeek/)을 만들었다. iOS·macOS·Windows 홈스크린/데스크톱 위젯에 코드가 상시 떠 있고, 탭(클릭) 한 번이면 클립보드에 복사된다. 폰을 꺼낼 필요도, 앱을 열 필요도 없다. 이름 그대로 힐끗(peek) 보고 가져다 쓰는 게 전부다.

![데스크톱 앱에서 계정별 2FA 코드가 한눈에 나열되고 클릭 한 번으로 복사되는 화면](/images/posts/otpeek-cross-platform-otp/desktop.png)

## 사실은 두 번 갈아엎었다

솔직히 처음부터 이렇게 만든 건 아니다. 지금 구조에 오기까지 큰 재작성을 두 번 했다.

**첫 삽은 C#/.NET로 떴다.** 작년 12월, Windows는 WinUI 3로 짜고 macOS·iOS는 .NET MAUI로 붙여서 한 언어로 전부 커버하려 했다. 그런데 정작 시간을 잡아먹은 건 OTP 로직이 아니라 주변부였다. 클라우드 동기화를 붙이려고 OneDrive/Graph SDK와 씨름했고, QR 스캔은 ZXing으로, 패키징은 Windows App SDK로 — MAUI가 각 OS의 네이티브 감성을 제대로 못 내줘서 애플 쪽 화면이 계속 어정쩡했다. 그 시절 커밋 로그는 인증기의 본질과 무관한 SDK 삽질로 가득하다.

그래서 **애플만 네이티브 SwiftUI로 다시 짰다.** 화면은 확실히 좋아졌는데, 이번엔 더 나쁜 문제가 생겼다. 한때 같은 TOTP·볼트 로직이 **세 벌** 굴러다녔다. Apple용 Swift에 하나, Windows용 C#에 하나, 아직 다 걷어내지 못한 MAUI 잔재에 하나. 버그를 하나 고치면 세 군데를 고쳐야 했고, RFC 6238 테스트 벡터도 언어마다 따로 통과시켜야 했다. 인증기의 본질은 "정확한 숫자를 만들고 안전하게 보관하는" 것 딱 두 가지인데, 그 두 가지가 코드베이스 세 곳에 흩어져 있었다.

**두 번째 재작성은 올해 7월, 로직을 통째로 Rust 코어로 옮기는 것이었다.** 문서에는 점잖게 "big-bang migration"이라고 써놨는데, 그냥 큰맘 먹고 한 번에 다 뜯어고쳤다는 뜻이다.

## 빅뱅을 하려면 계약부터 얼려야 했다

세 벌의 로직을 한 벌로 합치는 작업을 조금씩 나눠서 하면 오히려 지옥이다. 옛 코드와 새 코어가 공존하는 동안 데이터 포맷이 어긋나면 볼트가 깨진다. 그래서 반대로 갔다. **먼저 계약(contract)을 못 박아 얼린 뒤, 애플·Windows·CLI를 동시에 새 코어 위로 올리는 방식**이다.

얼린 결정 중 몇 개는 이런 것들이다.

- **시간은 언제나 `i64` epoch 밀리초(UTC)로 표현한다.** 로컬 타임존이나 문자열 포맷이 경계를 넘나들지 못하게, 플랫폼 사이에 오가는 타임스탬프의 타입을 처음부터 하나로 고정했다.
- **비밀번호 변경은 볼트를 다시 암호화하지 않는다.** 마스터 키(뒤에 나온다)를 새 비밀번호로 다시 감싸기만 하면 된다. 계정 수백 개를 재암호화하지 않으니 즉시 끝난다.
- **키 로테이션은 v2로 미룬다.** 하고 싶은 건 많았지만, 계약을 얼리려면 "이번 버전에 안 넣는 것"도 똑같이 명시해야 병렬 작업자들이 헷갈리지 않는다.

이렇게 표면 사이의 규약을 얼려두니, 각 플랫폼 구현을 서로 기다리지 않고 여러 갈래로 나란히 밀어붙일 수 있었다. 실제로 후반부 작업은 아이콘/패키징, 위젯 런타임, 테마/레이아웃, CLI 검증 같은 워크스트림을 파일 소유권이 겹치지 않게 쪼개서 병렬로 돌렸다. 계약이 흔들리지 않는 한, 한쪽에서 화면을 그리는 동안 다른 쪽에서 COM 수명을 고쳐도 충돌하지 않는다.

## 하나의 뇌, 세 개의 껍데기

새 구조의 원칙은 단순했다. **로직은 Rust로 딱 한 번만 짜고, UI는 각 플랫폼 네이티브로 얇게 유지한다.**

OTP 알고리즘, 데이터 모델, 볼트 암호화, 동기화 — 이 모든 걸 Rust 워크스페이스 하나에 몰아넣었다.

```
core/crates/
  otpeek-core/    # TOTP/HOTP, otpauth:// URI, 구글 인증기 마이그레이션
  otpeek-vault/   # 암호화 볼트 컨테이너, 파일 I/O
  otpeek-sync/    # 동기화 엔진, 병합 규칙, WebDAV 백엔드
  otpeek-ffi/     # UniFFI 파사드 (Swift / C# 바인딩)
  otpeek-cli/     # otpeek 커맨드라인 클라이언트
```

그리고 이 Rust 코어를 [UniFFI](https://mozilla.github.io/uniffi-rs/)로 Swift와 C#에 노출시켰다. 애플 쪽은 XCFramework로 뽑아 SwiftUI 앱이 그대로 호출하고, Windows는 `uniffi-bindgen-cs`로 C# 바인딩을 생성해 WinUI 3 앱이 호출한다. 두 앱 모두 OTP·암호화·동기화 로직을 단 한 줄도 다시 구현하지 않는다. 각 플랫폼에 남은 건 순수하게 화면을 그리는 코드뿐이다.

![iOS 앱 화면. 데스크톱과 같은 계정 목록을 SwiftUI가 같은 Rust 코어를 호출해 그대로 보여준다](/images/posts/otpeek-cross-platform-otp/mobile.png)

이렇게 바꾸고 나니 버그를 고칠 때 Rust 쪽 한 군데만 고치면 됐다. 테스트도 마찬가지다. RFC 4226/6238의 테스트 벡터를 코어에서 한 번 통과시키면, Swift에서 부르든 C#에서 부르든 같은 코드가 나온다. 참고로 코어 안에서는 `std::time`을 절대 부르지 않고 시간을 항상 인자로 받는데(그 인자가 위에서 얼린 epoch 밀리초다), 이래야 특정 시각의 코드를 결정론적으로 테스트할 수 있기 때문이다.

## 위젯은 절대 KDF를 돌리면 안 된다

이 프로젝트에서 제일 재밌었던 설계 문제는 위젯이었다.

OTPeek의 핵심 가치는 "위젯에 코드가 상시 떠 있다"는 건데, 위젯은 태생적으로 제약이 심한 환경이다. 백그라운드에서 잠깐 깨어나 화면을 그리고 다시 잠든다. 여기서 무거운 연산을 돌리면 안 되고, 비밀번호를 물어볼 UI도 없다.

그런데 볼트는 암호화되어 있다. 코드를 만들려면 볼트를 열어야 하고, 볼트를 열려면 키가 필요하다. 이 키를 어떻게 위젯에 넘길 것인가가 문제였다.

여기서 키 구조를 두 겹으로 나눴다.

- **VMK (Vault Master Key)**: 볼트를 만들 때 한 번 생성되는 랜덤 32바이트. 실제로 볼트 내용을 암호화하는 키다.
- **KEK**: 마스터 비밀번호를 Argon2id(64 MiB, t=3)로 돌려 만든 키. VMK를 감싸는(wrap) 용도다.

그리고 **로컬에서는 감싸지 않은 raw VMK를 OS 키스토어(Keychain / DPAPI / Secret Service)에 직접 저장한다.** 위젯은 키스토어에서 VMK를 꺼내 곧바로 볼트를 열면 된다. 무거운 Argon2는 건드릴 일이 없다. 앞에서 "비밀번호 변경은 VMK를 다시 감싸기만 한다"고 했던 게 여기서 이어진다. VMK 자체는 그대로 있으니 키스토어에 저장된 값도 바꿀 필요가 없다.

Argon2id가 실제로 돌아가는 순간은 딱 정해져 있다. 볼트를 처음 만들 때, 새 기기를 등록할 때, 비밀번호를 바꿀 때, 그리고 동기화 블롭을 열 때. 이 네 경우 모두 사용자가 비밀번호를 입력하는 "앱"의 맥락이지, 위젯이 아니다. 그래서 아키텍처 문서 곳곳에 이 문장을 못 박아뒀다. **"Widgets never run the KDF."** 위젯의 반응성과 볼트의 보안을 둘 다 지키려면 이 경계가 무너지면 안 됐다.

대신 정직하게 짚고 넘어갈 트레이드오프도 있다. 위젯은 별도 인증 없이 코드를 보여준다. 잠금 화면에 위젯을 올려두면 폰을 안 열고도 코드가 보인다는 뜻이다. 편의를 위한 의도된 설계지만, 계정별로 생체인증을 걸 수 있는 옵션은 앞으로 넣어야 할 숙제로 남겨뒀다.

## 파일 하나로 저장하고, 백업하고, 동기화한다

볼트를 설계하면서 마음에 들었던 부분이 하나 더 있다. `.otpvault` 파일 포맷 하나가 세 가지 역할을 전부 한다.

- 로컬에 저장되는 볼트
- 내보내기(export)한 백업 파일
- 원격 서버에 올라가는 동기화 블롭

전부 같은 포맷이다. AES-256-GCM으로 암호화되어 있어서, 동기화 백엔드는 어떤 경우에도 암호문 바이트만 본다. 서버가 iCloud든 WebDAV든 상관없이 평문은 절대 밖으로 나가지 않는다.

동기화 백엔드는 "불투명한 바이트를 옮기는" 멍청한 역할만 한다. 애플은 CloudKit, 나머지는 WebDAV(Nextcloud 같은)를 쓰는데, 이 백엔드들은 그저 암호화된 덩어리를 주고받을 뿐이다. **정작 어려운 병합 로직은 전부 Rust 코어에 있다.** 계정별로 `updated_at`이 큰 쪽이 이기는 last-writer-wins, 삭제는 tombstone으로 전파, 그리고 HOTP 카운터만은 예외적으로 양쪽의 최댓값을 취한다 — 카운터를 재사용하면 코드가 깨지니까. 이 규칙을 백엔드마다 다시 구현했으면 지옥이었을 텐데, 코어에 한 번 넣어두니 애플이든 WebDAV든 똑같이 동작한다.

## macOS 볼트를 Windows에 그냥 복사하면 안 열린다

여기서 키를 두 겹으로 나눈 설계가 진짜로 살아 있다는 걸 확인한 순간이 있었다. macOS에서 쓰던 볼트를 Windows로 옮겨 테스트하려고, 처음엔 순진하게 볼트 파일을 그대로 복사했다. 안 열렸다.

당연한 결과였다. 볼트 파일 옆에 있는 원본 raw VMK는 **macOS Keychain**과 **Windows DPAPI**라는 서로 다른 OS 키스토어에 묶여 있다. 키 저장 방식이 다르니 파일만 옮겨서는 위젯도 앱도 볼트를 열 수 없다. 이건 버그가 아니라 설계가 의도한 대로 작동한 것이다 — raw 키는 그 기기의 OS 보안 저장소를 떠나지 않는다.

정답은 이미 만들어 둔 길에 있었다. 원본 파일을 복사하는 대신, macOS에서 암호화된 `.otpvault`로 **export**하고 Windows의 빈 볼트에 **import(병합)** 하는 것. export 파일은 마스터 비밀번호로 다시 감싼 자기완결형 포맷이라, 목적지 기기에서 비밀번호 한 번만 넣으면 그 기기의 DPAPI에 새로 VMK가 자리잡는다. Keychain과 DPAPI의 차이는 이 경로가 알아서 흡수한다. 불편해 보이지만, 이게 바로 "raw 키는 기기 밖으로 안 나간다"는 원칙이 실제로 지켜지고 있다는 증거였다.

## 기기를 늘릴 때가 진짜 시험대다

인증기 앱을 옮겨본 사람은 안다. 폰을 바꾸거나 기기를 하나 더 붙일 때가 제일 짜증 난다. 계정을 하나씩 다시 등록하거나, QR을 다시 찍거나 하다 보면 몇 개는 꼭 빠뜨린다.

그래서 "한 번 등록하고 어디서든 쓴다"를 목표로 잡았다. 원리는 앞의 볼트 구조에서 그대로 나온다. 새 기기는 암호화된 블롭을 받아와서 마스터 비밀번호를 딱 한 번 입력한다. 그러면 그 기기의 키스토어에 VMK가 자리잡고, 이후로는 비밀번호 없이 조용히 열리며 동기화도 알아서 병합된다.

- **애플 기기끼리(같은 Apple ID)** — 첫 기기에서 iCloud 동기화를 켜면, 다른 기기는 "iCloud에서 복원"을 고르고 마스터 비밀번호만 한 번 넣으면 된다.
- **플랫폼을 넘나들거나 직접 서버를 쓸 때** — 모든 기기를 같은 WebDAV URL로 가리키게 한다. 서버에는 암호화된 덩어리만 쌓인다.
- **서버가 아예 싫을 때** — `.otpvault`로 내보내서 더블클릭(Windows/macOS)하거나 AirDrop(애플)으로 보내면, 앱이 곧장 가져오기 화면으로 열린다. 백업 비밀번호만 넣으면 끝.

구글 인증기에서 넘어오는 것도 신경 썼다. 구글 인증기의 내보내기 QR(`otpauth-migration://`)을 그대로 읽어서 한 번에 여러 계정을 가져온다. 예전에 쓰던 v1 백업 포맷도 import 전용으로 남겨뒀다 — 옛날 데이터를 버리게 하고 싶지 않았다.

## Windows는 늘 막내였다

세 플랫폼 중 Windows는 항상 한 발 늦었다. 첫 재작성 때 애플을 SwiftUI로 밀어주는 동안 Windows는 뒤로 밀렸고, Rust 코어로 합친 뒤에도 macOS/iOS 대비 화면·아이콘·위젯이 한참 뒤떨어져 있었다. 이걸 따라잡는 과정에서 두 번, 제대로 헤맸다.

**첫 번째는 시작하자마자 죽는 앱이었다.** 실제 Windows 머신에서 Release 빌드는 오류 0개로 통과했는데, 실행하면 창이 뜨지도 않고 즉시 종료됐다. 이벤트 로그를 까보니 `Microsoft.UI.Xaml.dll` 안에서 `STATUS_STOWED_EXCEPTION`(`0xc000027b`)으로 시작 직후 크래시. 처음엔 이번 UI 개선이 만든 회귀인 줄 알았다. 그래서 기준 커밋을 **별도 worktree에 따로 빌드**해서 "예전 코드도 똑같이 죽는지"부터 비교했다. 똑같이 죽었다. 회귀가 아니라 원래부터 WinUI 시작 경로가 실행 불가능했던 것이다. 로그 한 줄조차 안 남는 걸 보니 `new App()`보다도 앞, WinUI 런타임 기동부에서 죽고 있었다.

한참을 SDK 버전, 생성된 엔트리포인트, `resources.pri`, self-contained 배포 구성까지 파고들었는데 — **진짜 원인은 완벽한 레드헤링이었다.** 나는 원격 머신에 SSH로 붙어서 앱을 실행하고 있었는데, WinUI는 SSH가 붙는 **세션 0(서비스 세션)** 에서는 초기화 자체가 불가능했다. 실제로 로그인된 데스크톱 **세션 1**에서 같은 빌드를 띄우니 창이 멀쩡히 뜨고 응답했다. 앱은 처음부터 정상이었고, 죽은 건 내가 앱을 잘못된 자리에서 실행했기 때문이었다. (그 와중에 진짜 코드 버그도 하나 있긴 했다. `TypeComboBox`의 이벤트가 XAML 초기화 도중 너무 일찍 발생해 아직 생성되지 않은 컨트롤을 건드리던 문제라, 이벤트 연결을 `InitializeComponent()` 뒤로 옮겨 고쳤다.)

**두 번째는 트레이 메뉴였다.** 트레이 아이콘을 우클릭하면 뜨는 팝업 메뉴를 macOS 수준으로 예쁘게 만들고 싶었다. favicon도 넣고, 남은 시간 카운트다운도 넣고, 실시간으로 갱신되게. 그렇게 만들었더니 우클릭할 때마다 별도의 WinUI 창과 계정 ViewModel 10개, favicon 디코딩, 타이머 여러 개를 새로 생성하느라 반응이 굼뜨고 버벅였다. 우클릭 메뉴가 우클릭 메뉴답지 않게 무거웠다.

결국 방향을 틀었다. `H.NotifyIcon`의 네이티브 메뉴는 Win32 `TrackPopupMenuEx`를 쓰기 때문에 반응성은 최고지만 **이미지 아이콘을 지원하지 않는다.** 그래서 **속도와 안정성을 위해 우클릭 메뉴에서는 favicon과 실시간 애니메이션을 의도적으로 포기했다.** 대신 계정명은 적당히 자르고, 오른쪽에 코드와 남은 초를 정렬해 인지가 잘 되게 했다. favicon과 1초 카운트다운이 있는 화려한 라이브 팝업은 좌클릭 쪽에만 남겼다. 원래 프로젝트 사양도 "좌클릭=라이브 팝업, 우클릭=가벼운 컨텍스트 메뉴"로 나뉘어 있었으니, 결과적으로 제자리를 찾은 셈이다. 위젯도 favicon을 각 행에 실어 표시하고, "click to copy" 같은 안내 문구를 걷어내 그 공간에 계정을 더 많이(중형 5개, 대형 8개) 보여주도록 다듬었다.

## 터미널에서도 쓰고 싶었다

로직을 Rust로 몰아넣고 나니 공짜로 딸려온 게 있었다. 앱이나 위젯이 아니라 **커맨드라인 클라이언트**다.

나는 하루의 대부분을 터미널에서 보낸다. 서버에 SSH로 붙어서 뭘 하다가 2FA 코드가 필요할 때, GUI 앱을 찾는 것도 번거롭다. 그래서 코어를 그대로 쓰는 `otpeek` 바이너리를 만들었다.

```bash
otpeek init                     # 볼트 생성
otpeek add 'otpauth://totp/GitHub:me?secret=...&issuer=GitHub'
otpeek code github --copy       # 코드 생성 후 클립보드 복사
```

`--watch`를 붙이면 카운트다운과 함께 TOTP가 갱신되는 걸 터미널에서 볼 수 있고, `--json`으로 스크립트에 물릴 수도 있다. 헤드리스 리눅스에서도 환경변수로 비밀번호를 넘기면 키스토어 없이 돌아간다. CLI는 코어·볼트·동기화 크레이트에만 의존하고 FFI 레이어는 거치지 않는데, 이것도 로직이 한 곳에 모여 있었기에 얻은 이득이다. 애초에 계약을 얼려두고 애플·Windows와 나란히 CLI를 밀어붙일 수 있었던 것도 같은 이유다.

여담으로, Intel 맥용 CLI 바이너리를 Apple Silicon 러너에서 크로스컴파일하는 CI를 맞추느라 조금 삽질했다. 요즘 GitHub Actions의 맥 러너가 arm64라 x86_64 타깃을 따로 붙여줘야 했다.

## 마무리

지금 OTPeek은 [App Store](https://apps.apple.com/app/otpeek/id6787845951)에 올라가 있고, [랜딩 페이지](https://jiun.dev/OTPeek/)와 개인정보 처리방침, 서포트 페이지까지 붙여서 배포된 상태다. 위젯, 생체인증 잠금, QR 가져오기(구글 인증기 마이그레이션 포함), 폴더/즐겨찾기, iCloud·WebDAV 동기화까지 실제로 매일 내가 쓰는 도구가 됐다. Windows도 이제 실제 머신에서 앱·위젯·트레이가 다 돌아가는 걸 확인했다.

물론 아직 다듬을 곳은 많다. 클립보드에 복사한 코드를 일정 시간 뒤 자동으로 지우는 기능은 아직 없고(수동 관리 중이다), 위젯 계정별 생체인증도 미완이다. 얼려둔 계약에서 v2로 미뤄둔 키 로테이션도 여전히 숙제로 남아 있다. 앱스토어에 올리면서 팁 자(tip jar)랑 광고 제거 IAP 같은 수익화도 붙여봤는데, 이 부분은 솔직히 실험에 가깝다.

그래도 처음 만들 때 세웠던 원칙 하나는 끝까지 지켰다. **인증기의 본질인 "정확한 코드"와 "안전한 보관"은 딱 한 번만 구현한다.** 세 플랫폼에서 같은 로직을 세 번 짜던 시절로는 돌아가고 싶지 않다. 그리고 그 위에서 진짜 하고 싶었던 것 — 폰을 꺼내지 않고 코드를 힐끗 보고 가져다 쓰는 것 — 은 이제 잘 된다.
