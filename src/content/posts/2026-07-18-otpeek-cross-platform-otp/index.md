# 2FA 코드 하나 보려고 폰을 꺼내는 게 싫어서 만든 OTPeek

## 들어가며

데스크톱 앞에 앉아서 로그인을 하는데 2FA 코드를 넣으라고 한다. 그럼 나는:

1. 주머니에서 폰을 꺼내고
2. 폰을 잠금 해제하고
3. 인증기 앱을 열고
4. 수십 개 계정 중에 해당 계정을 찾고
5. 6자리를 눈으로 읽고
6. 코드가 만료되기 전에 키보드로 옮겨 친다

이걸 하루에 몇 번씩 한다. 손 뻗으면 닿는 곳에 컴퓨터가 있는데, 굳이 폰을 꺼내서 눈으로 숫자를 읽고 다시 타이핑하는 이 왕복이 매번 거슬렸다. 코드는 이미 화면 어딘가에 항상 떠 있으면 되는 거 아닌가? 위젯 하나에 코드가 보이고, 탭 한 번에 복사되면 끝날 일이었다.

그래서 [OTPeek](https://jiun.dev/OTPeek/)을 만들었다. iOS·macOS·Windows 홈스크린/데스크톱 위젯에 코드가 상시 떠 있고, 탭(클릭) 한 번이면 클립보드에 복사된다. 폰을 꺼낼 필요도, 앱을 열 필요도 없다. 이름 그대로 힐끗(peek) 보고 가져다 쓰는 게 전부다.

## 사실은 두 번 만들었다

솔직히 처음부터 이렇게 만든 건 아니다.

작년 12월에 처음 손댔을 때는 C#/.NET 기반이었다. WinUI 3로 Windows 앱을 짜고, macOS·iOS는 .NET MAUI로 붙이고, OTP 생성 로직과 볼트 암호화, 클라우드 동기화를 각각 구현했다. 몇 주 동안은 그럭저럭 굴러갔다. 그런데 문제가 하나 있었다. **같은 로직을 플랫폼마다 다시 짜고 있었다.**

TOTP 계산 코드가 Swift에 하나, C#에 하나. 볼트 암호화가 또 각자. 버그를 하나 고치면 두 군데를 고쳐야 했고, RFC 6238 테스트 벡터를 통과하는지도 각 언어로 따로 검증해야 했다. OneDrive/Graph SDK, ZXing, Windows App SDK 같은 것들과 씨름한 커밋이 그 시절 로그에 잔뜩 남아 있다. 인증기 앱의 본질은 "정확한 숫자를 만들고, 안전하게 보관하는" 것 딱 두 가지인데, 정작 그 두 가지가 코드베이스 세 군데에 흩어져 있었다.

그래서 올해 7월, 전부 갈아엎었다. 문서에는 점잖게 "big-bang migration"이라고 써놨는데, 그냥 큰맘 먹고 한 번에 다 뜯어고쳤다는 뜻이다.

## 하나의 뇌, 세 개의 껍데기

새 구조의 원칙은 단순했다. **로직은 Rust로 딱 한 번만 짜고, UI는 각 플랫폼 네이티브로 얇게 유지한다.**

OTP 알고리즘, 데이터 모델, 볼트 암호화, 동기화 — 이 모든 걸 Rust 워크스페이스 하나에 몰아넣었다.

```
core/crates/
  otpeek-core/    # TOTP/HOTP, otpauth:// URI, 구글 인증기 마이그레이션
  otpeek-vault/   # 암호화 볼트 컨테이너, 파일 I/O
  otpeek-sync/    # 동기화 엔진, 병합 규칙, WebDAV 백엔드
  otpeek-ffi/     # UniFFI 파사드 (Swift / C# 바인딩)
  otpeek-cli/     # otpeek 커맨드라인 클라이언트
```

그리고 이 Rust 코어를 [UniFFI](https://mozilla.github.io/uniffi-rs/)로 Swift와 C#에 노출시켰다. 애플 쪽은 XCFramework로 뽑아 SwiftUI 앱이 그대로 호출하고, Windows는 `uniffi-bindgen-cs`로 C# 바인딩을 생성해 WinUI 3 앱이 호출한다. 두 앱 모두 OTP·암호화·동기화 로직을 단 한 줄도 다시 구현하지 않는다. 각 플랫폼에 남은 건 순수하게 화면을 그리는 코드뿐이다.

이렇게 바꾸고 나니 버그를 고칠 때 Rust 쪽 한 군데만 고치면 됐다. 테스트도 마찬가지다. RFC 4226/6238의 테스트 벡터를 코어에서 한 번 통과시키면, Swift에서 부르든 C#에서 부르든 같은 코드가 나온다. 참고로 코어 안에서는 `std::time`을 절대 부르지 않고 시간을 항상 인자로 받는데, 이래야 특정 시각의 코드를 결정론적으로 테스트할 수 있기 때문이다.

## 위젯은 절대 KDF를 돌리면 안 된다

이 프로젝트에서 제일 재밌었던 설계 문제는 위젯이었다.

OTPeek의 핵심 가치는 "위젯에 코드가 상시 떠 있다"는 건데, 위젯은 태생적으로 제약이 심한 환경이다. 백그라운드에서 잠깐 깨어나 화면을 그리고 다시 잠든다. 여기서 무거운 연산을 돌리면 안 되고, 비밀번호를 물어볼 UI도 없다.

그런데 볼트는 암호화되어 있다. 코드를 만들려면 볼트를 열어야 하고, 볼트를 열려면 키가 필요하다. 이 키를 어떻게 위젯에 넘길 것인가가 문제였다.

여기서 키 구조를 두 겹으로 나눴다.

- **VMK (Vault Master Key)**: 볼트를 만들 때 한 번 생성되는 랜덤 32바이트. 실제로 볼트 내용을 암호화하는 키다.
- **KEK**: 마스터 비밀번호를 Argon2id(64 MiB, t=3)로 돌려 만든 키. VMK를 감싸는(wrap) 용도다.

그리고 **로컬에서는 감싸지 않은 raw VMK를 OS 키스토어(Keychain / DPAPI / Secret Service)에 직접 저장한다.** 위젯은 키스토어에서 VMK를 꺼내 곧바로 볼트를 열면 된다. 무거운 Argon2는 건드릴 일이 없다.

Argon2id가 실제로 돌아가는 순간은 딱 정해져 있다. 볼트를 처음 만들 때, 새 기기를 등록할 때, 비밀번호를 바꿀 때, 그리고 동기화 블롭을 열 때. 이 네 경우 모두 사용자가 비밀번호를 입력하는 "앱"의 맥락이지, 위젯이 아니다. 그래서 아키텍처 문서 곳곳에 이 문장을 못 박아뒀다. **"Widgets never run the KDF."** 위젯의 반응성과 볼트의 보안을 둘 다 지키려면 이 경계가 무너지면 안 됐다.

대신 정직하게 짚고 넘어갈 트레이드오프도 있다. 위젯은 별도 인증 없이 코드를 보여준다. 잠금 화면에 위젯을 올려두면 폰을 안 열고도 코드가 보인다는 뜻이다. 편의를 위한 의도된 설계지만, 계정별로 생체인증을 걸 수 있는 옵션은 앞으로 넣어야 할 숙제로 남겨뒀다.

## 파일 하나로 저장하고, 백업하고, 동기화한다

볼트를 설계하면서 마음에 들었던 부분이 하나 더 있다. `.otpvault` 파일 포맷 하나가 세 가지 역할을 전부 한다.

- 로컬에 저장되는 볼트
- 내보내기(export)한 백업 파일
- 원격 서버에 올라가는 동기화 블롭

전부 같은 포맷이다. AES-256-GCM으로 암호화되어 있어서, 동기화 백엔드는 어떤 경우에도 암호문 바이트만 본다. 서버가 iCloud든 WebDAV든 상관없이 평문은 절대 밖으로 나가지 않는다.

동기화 백엔드는 "불투명한 바이트를 옮기는" 멍청한 역할만 한다. 애플은 CloudKit, 나머지는 WebDAV(Nextcloud 같은)를 쓰는데, 이 백엔드들은 그저 암호화된 덩어리를 주고받을 뿐이다. **정작 어려운 병합 로직은 전부 Rust 코어에 있다.** 계정별로 `updated_at`이 큰 쪽이 이기는 last-writer-wins, 삭제는 tombstone으로 전파, 그리고 HOTP 카운터만은 예외적으로 양쪽의 최댓값을 취한다 — 카운터를 재사용하면 코드가 깨지니까. 이 규칙을 백엔드마다 다시 구현했으면 지옥이었을 텐데, 코어에 한 번 넣어두니 애플이든 WebDAV든 똑같이 동작한다.

## 기기를 늘릴 때가 진짜 시험대다

인증기 앱을 옮겨본 사람은 안다. 폰을 바꾸거나 기기를 하나 더 붙일 때가 제일 짜증 난다. 계정을 하나씩 다시 등록하거나, QR을 다시 찍거나 하다 보면 몇 개는 꼭 빠뜨린다.

그래서 "한 번 등록하고 어디서든 쓴다"를 목표로 잡았다. 원리는 앞의 볼트 구조에서 그대로 나온다. 새 기기는 암호화된 블롭을 받아와서 마스터 비밀번호를 딱 한 번 입력한다. 그러면 그 기기의 키스토어에 VMK가 자리잡고, 이후로는 비밀번호 없이 조용히 열리며 동기화도 알아서 병합된다.

- **애플 기기끼리(같은 Apple ID)** — 첫 기기에서 iCloud 동기화를 켜면, 다른 기기는 "iCloud에서 복원"을 고르고 마스터 비밀번호만 한 번 넣으면 된다.
- **플랫폼을 넘나들거나 직접 서버를 쓸 때** — 모든 기기를 같은 WebDAV URL로 가리키게 한다. 서버에는 암호화된 덩어리만 쌓인다.
- **서버가 아예 싫을 때** — `.otpvault`로 내보내서 더블클릭(Windows/macOS)하거나 AirDrop(애플)으로 보내면, 앱이 곧장 가져오기 화면으로 열린다. 백업 비밀번호만 넣으면 끝.

구글 인증기에서 넘어오는 것도 신경 썼다. 구글 인증기의 내보내기 QR(`otpauth-migration://`)을 그대로 읽어서 한 번에 여러 계정을 가져온다. 예전에 쓰던 v1 백업 포맷도 import 전용으로 남겨뒀다 — 옛날 데이터를 버리게 하고 싶지 않았다.

## 터미널에서도 쓰고 싶었다

로직을 Rust로 몰아넣고 나니 공짜로 딸려온 게 있었다. 앱이나 위젯이 아니라 **커맨드라인 클라이언트**다.

나는 하루의 대부분을 터미널에서 보낸다. 서버에 SSH로 붙어서 뭘 하다가 2FA 코드가 필요할 때, GUI 앱을 찾는 것도 번거롭다. 그래서 코어를 그대로 쓰는 `otpeek` 바이너리를 만들었다.

```bash
otpeek init                     # 볼트 생성
otpeek add 'otpauth://totp/GitHub:me?secret=...&issuer=GitHub'
otpeek code github --copy       # 코드 생성 후 클립보드 복사
```

`--watch`를 붙이면 카운트다운과 함께 TOTP가 갱신되는 걸 터미널에서 볼 수 있고, `--json`으로 스크립트에 물릴 수도 있다. 헤드리스 리눅스에서도 환경변수로 비밀번호를 넘기면 키스토어 없이 돌아간다. CLI는 코어·볼트·동기화 크레이트에만 의존하고 FFI 레이어는 거치지 않는데, 이것도 로직이 한 곳에 모여 있었기에 얻은 이득이다.

여담으로, Intel 맥용 CLI 바이너리를 Apple Silicon 러너에서 크로스컴파일하는 CI를 맞추느라 조금 삽질했다. 요즘 GitHub Actions의 맥 러너가 arm64라 x86_64 타깃을 따로 붙여줘야 했다.

## 마무리

지금 OTPeek은 [App Store](https://apps.apple.com/app/otpeek/id6787845951)에 올라가 있고, [랜딩 페이지](https://jiun.dev/OTPeek/)와 개인정보 처리방침, 서포트 페이지까지 붙여서 배포된 상태다. 위젯, 생체인증 잠금, QR 가져오기(구글 인증기 마이그레이션 포함), 폴더/즐겨찾기, iCloud·WebDAV 동기화까지 실제로 매일 내가 쓰는 도구가 됐다.

물론 아직 다듬을 곳은 많다. 클립보드에 복사한 코드를 일정 시간 뒤 자동으로 지우는 기능은 아직 없고(수동 관리 중이다), 위젯 계정별 생체인증도 미완이다. 앱스토어에 올리면서 팁 자(tip jar)랑 광고 제거 IAP 같은 수익화도 붙여봤는데, 이 부분은 솔직히 실험에 가깝다.

그래도 처음 만들 때 세웠던 원칙 하나는 끝까지 지켰다. **인증기의 본질인 "정확한 코드"와 "안전한 보관"은 딱 한 번만 구현한다.** 세 플랫폼에서 같은 로직을 세 번 짜던 시절로는 돌아가고 싶지 않다. 그리고 그 위에서 진짜 하고 싶었던 것 — 폰을 꺼내지 않고 코드를 힐끗 보고 가져다 쓰는 것 — 은 이제 잘 된다.
